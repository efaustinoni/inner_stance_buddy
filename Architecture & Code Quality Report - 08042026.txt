# Architecture & Code Quality Report: Inner Stance Buddy

## Overall Grade: B (80/100)

Inner Stance Buddy is a well-crafted personal coaching PWA built on React 18, TypeScript, Vite, and Supabase. The codebase shows the hallmarks of a thoughtful solo/small-team project that has been actively improved: a clean component hierarchy, a dedicated service layer, meaningful tests, and unusually thorough documentation. The main growth areas are a single overweight service file that should be split, a few sequential database call chains that could be parallelized, the absence of a CI pipeline, and some leftover development artifacts that didn't get cleaned up before the review.

## Detailed Analysis

### 1. Architecture & Modularization — B (82/100)

*What this measures: How well the codebase is structured into clear, independent modules with well-defined boundaries and separation of concerns.*

The folder layout tells a coherent story at a glance: `src/components/{exercises,layout,pages,profile,ui}` groups UI by feature and layer, `src/lib/` holds all data-access and utility code, and `src/hooks/` holds extracted React hooks. This is a well-thought-out structure for a project of this size. The separation between the "how to display" world (components) and the "how to talk to the database" world (lib/exerciseService.ts, lib/legalService.ts, lib/supabase.ts) is consistently maintained. No component reaches into Supabase directly — everything goes through the service layer, which is the right pattern.

The `src/lib/timezone/` sub-package is a small highlight: it has its own `types.ts`, `index.ts` barrel, and three focused utility files (`detection.ts`, `formatting.ts`, `grouping.ts`), plus a `data/` folder for static lookup tables. This is textbook modular design for a standalone utility.

The primary structural concern is `src/lib/exerciseService.ts` at 865+ lines, covering quarters, weeks, questions, answers, bulk import, AI image extraction, progress trackers, and check-ins. This one file accounts for the bulk of the app's data logic. A natural split would be `weekService.ts`, `questionService.ts`, `answerService.ts`, and `trackerService.ts` — each cohesive and much easier to reason about or mock in isolation.

**Strengths:**
- Feature-based component subdirectories (`exercises/`, `profile/`, `pages/`) make navigation intuitive for a new developer
- `src/hooks/useAuth.ts` and `useLegalStatus.ts` were recently extracted from `App.tsx`, showing active architectural improvement (Phase 2 in changelog)
- The `App.tsx` routing file is clean and under 135 lines — no business logic bleeds in

**Issues:**
- `src/lib/exerciseService.ts` is an 865-line monolith mixing quarter CRUD, week CRUD, question CRUD, answer CRUD, bulk import, AI extraction, tracker CRUD, and check-in CRUD — it should be split by domain
- `PowerPage.tsx` (540 lines) and `DashboardPage.tsx` (~500 lines) are among the larger page components and could each have a custom hook extracted to reduce their local state surface

### 2. Shared Components & Reuse — B (78/100)

*What this measures: How effectively common patterns, utilities, and components are extracted and reused rather than duplicated across the codebase.*

The `src/components/ui/` directory functions as a mini design system: `Button.tsx` covers five variants and three sizes with a loading state; `Card.tsx` provides four variants plus `CardHeader`, `CardTitle`, `CardContent`, and `CardFooter` sub-components; `Avatar.tsx` handles photo, initials, and fallback states. These are all real, reusable abstractions, not one-off wrappers.

Barrel `index.ts` files in `exercises/`, `layout/`, `pages/`, `profile/`, and `ui/` keep import lines tidy throughout the codebase. The toast system (`lib/toast.ts` + `components/ui/Toast.tsx`) is a lightweight custom implementation that avoids pulling in a heavy dependency.

A recurring pattern worth noting is the inline loading spinner: `<div className="w-8 h-8 border-4 border-accent-blue border-t-transparent rounded-full animate-spin" />` appears in `App.tsx`, `DashboardPage.tsx`, `ExerciseQuestionCard.tsx`, and several other places. A `<Spinner />` component would be a three-line extraction that eliminates this repetition. Similarly, there is no shared `<LoadingPage>` or `<EmptyState>` component, so those patterns are implemented ad-hoc each time they're needed.

**Strengths:**
- `Button.tsx` and `Card.tsx` are genuinely reusable design-system components, not page-specific wrappers
- The `ErrorBoundary` and `Toast` system are app-wide, single-instance singletons properly placed in `ui/`
- The timezone utility module is entirely self-contained and importable from a clean barrel

**Issues:**
- Inline loading spinner duplicated in at least 5 places — should be a `<Spinner />` component
- `fetchDashboardData` and `fetchUserWeeks` both contain nearly identical join-and-normalize logic for the `exercise_quarters → quarter_label` mapping (lines 897–902 and 122–129 of exerciseService.ts)

### 3. Code Quality & Readability — B (80/100)

*What this measures: How clean, consistent, and understandable the code is — whether someone new could read it and follow what's happening.*

The codebase uses TypeScript throughout with well-typed interfaces (`ExerciseWeek`, `ExerciseQuestion`, `ExerciseAnswer`, `QuestionWithAnswer`, `WeekWithQuestions`, etc.) all defined and exported from the service file. Prettier is configured and enforced via a Husky pre-commit hook, so formatting is consistent. ESLint with `typescript-eslint` and `react-hooks` plugins catches common mistakes. Conventional commit messages are enforced with commitlint. File headers with `Created` and `Last Updated` dates are a nice documentation habit.

The main readability concern is debug logging left in production-path code. `bulkImportQuestions` in `exerciseService.ts` contains five `console.log` statements at the service layer (e.g., `[BulkImport] Received questions:`, `[BulkImport] Deleted N existing questions`, `[BulkImport] Inserted N questions`). These are verbose enough to produce noisy browser consoles for every import operation. They should be gated behind a `DEBUG` flag or removed.

`src/lib/appConfig.ts` still has placeholder branding — `appName: 'Your App Name'` and `appInitials: 'YAN'` — which is a minor but visible signal that the project's own configuration hasn't been finalized. It also introduces a subtle mismatch: the README and CHANGELOG refer to the app as "Inner Stance Buddy" but the running app would display "Your App Name."

**Strengths:**
- TypeScript interfaces are well-structured, exported, and reused across component props — no `any` types observed
- Prettier + ESLint + Husky stack enforces consistent style automatically
- Consistent use of named exports (not default everywhere) makes imports greppable and refactoring-safe

**Issues:**
- Five `console.log` statements in `bulkImportQuestions` will emit debug output in production
- `appConfig.ts` contains placeholder branding (`'Your App Name'`, `'YAN'`) that was never replaced
- `fetchTrackerWithCheckIns` uses the `!` non-null assertion on `question!` and `week!` on line 769-770 when those values could legitimately be undefined if data is deleted mid-session

### 4. Error Handling & Graceful Degradation — B+ (83/100)

*What this measures: How well the application handles unexpected situations — bad input, failed services, edge cases — without crashing or exposing internals.*

Error handling was comprehensively improved in the Phase 1 architecture pass (CHANGELOG entry `2026-04-07`). The additions show a mature approach: an `ErrorBoundary` class component wraps the entire app so unhandled render errors show a friendly "Something went wrong" screen rather than a blank page. A lightweight custom toast system fires user-visible error notifications on every critical write operation (save answer, delete week, bulk import, toggle check-in, etc.). `DashboardPage` has an explicit `loadError` state that shows a "Try Again" retry button rather than silently showing empty data.

All service functions follow a consistent contract: return `null`, `false`, or `[]` on error rather than throwing, which means components never need to wrap service calls in their own try-catch for normal error handling. Supabase env-var validation uses a fail-fast throw at module load time — a good pattern that surfaces misconfiguration immediately.

The one area where the safety net has gaps is `fetchTrackerWithCheckIns`, which chains four sequential Supabase calls and then composes the result with non-null assertions (`question!`, `week!`). If a question or week has been deleted between the tracker being created and this fetch running, the function would return an object with `undefined` values typed as non-null — a silent corruption that would cause downstream rendering errors.

**Strengths:**
- `ErrorBoundary` at app root provides a universal catch-all for render errors
- Every page-level data load has an explicit error state with user-visible feedback and retry affordance
- Supabase client initialization throws immediately if env vars are missing — no silent failure at runtime

**Issues:**
- `fetchTrackerWithCheckIns` uses `question!` and `week!` non-null assertions on data that may not exist (lines 769-770)
- The `ErrorBoundary` shows `error.message` directly to the user in a monospace block — this could expose internal error details in production; consider filtering this behind a dev-mode flag

### 5. Testing — B (77/100)

*What this measures: Whether there is a meaningful testing strategy — not just whether tests exist, but whether they actually catch bugs and cover important behavior.*

The testing strategy is notably strong for a project of this size. There is an almost perfect 1:1 pairing of `.test.tsx` files alongside every component file. The mock strategy is well-structured: `src/lib/__mocks__/` contains hand-rolled mocks for `supabase`, `exerciseService`, `legalService`, and `crypto`, all colocated with the modules they mock. The `AuthPage.test.tsx` reviewed is a good example of behavioral tests — it checks button enable/disable states, Supabase call invocation, and error message display, not just that the component renders without crashing.

Playwright E2E tests in `e2e/` cover the unauthenticated flow (sign-in, sign-up, forgot-password, root redirect) with proper Supabase mocking via a `helpers/mock-supabase` utility, and an `auth.spec.ts` for the authenticated path.

The coverage thresholds in `vite.config.ts` are modest — 50% lines, 44% functions, 48% branches — and `src/lib/**` is entirely excluded from coverage. This means the service layer (exerciseService.ts, legalService.ts, crypto.ts), which contains the most complex and risk-prone logic, has no measured test coverage at all. The service layer is where bugs are most costly and where unit tests would add the most value.

**Strengths:**
- Near 1:1 component-to-test file ratio — no untested components
- `__mocks__` directory with proper module mocks enables isolated component testing
- Playwright E2E with Supabase mocking catches integration regressions without needing a live backend

**Issues:**
- `src/lib/**` is excluded from Vitest coverage — the service layer (where complex query logic lives) has no measured unit test coverage
- Coverage thresholds are set relatively low (50% lines); raising to 70%+ and including `lib/` would meaningfully strengthen the safety net
- No tests for `useAuth` or `useLegalStatus` hooks, which manage auth state and legal gating logic for the whole app

### 7. Code Efficiency & Performance — B (77/100)

*What this measures: Whether the code avoids obvious performance anti-patterns and handles resources responsibly.*

The most data-intensive operation in the app — `fetchDashboardData` — is well-optimized: after the initial waterfall (weeks → questions → answers + trackers in parallel), all joins are done in-memory using `Map` lookups, giving O(1) lookup per question rather than nested array scans. The `filteredQuestions` and `groupedByQuarterAndWeek` computations in `DashboardPage` are both wrapped in `useMemo` with correct dependency arrays, so re-renders don't re-run expensive array operations unnecessarily.

`fetchWeekWithQuestions` was recently parallelized with `Promise.all` (per the changelog), reducing three serial round-trips to two. The answer batch-insert optimization in `copyWeekToQuarter` is another good sign that performance is being actively considered.

The remaining performance gap is `fetchTrackerWithCheckIns`, which makes four sequential database queries (tracker → question → week → check-ins → answer), each waiting on the prior one. The question/week/check-ins/answer fetches are all independent once the tracker is known and could be run in a single `Promise.all`, cutting the round-trip count roughly in half.

**Strengths:**
- `fetchDashboardData` uses `Map` for all in-memory joins — no O(n²) array scans
- `useMemo` correctly applied for expensive derived state in `DashboardPage`
- `fetchWeekWithQuestions` parallelized (changelog documents the improvement from 3 to 2 round-trips)

**Issues:**
- `fetchTrackerWithCheckIns` has 4 sequential round-trips that could be reduced to 2 with `Promise.all`
- `saveAnswer` and `toggleCheckIn` each do a read-then-write (select existing, then insert or update) — these could use an upsert operation instead to save a round-trip
- `bulkImportQuestions` intentionally inserts questions sequentially for ordering reliability, but the comment explaining this trade-off is helpful — the approach is defensible given the constraint

### 8. Security — A- (88/100)

*What this measures: Whether basic security practices are followed — secrets management, input sanitization, auth patterns, and dependency hygiene.*

Security practices here are notably above average for a project at this stage. Supabase credentials are loaded exclusively from environment variables, `.env` is in `.gitignore`, and the Supabase client throws immediately if keys are missing. Row Level Security is documented as enforced on all tables. The Phase 3 architecture improvements added a `Content-Security-Policy` header to `public/_headers` restricting script-src, style-src, and connect-src. CORS in the three Supabase Edge Functions was tightened to the `SITE_URL` environment variable. Security question answers are hashed client-side before transmission, and the recent addition of a per-user random 32-byte salt (via `crypto.getRandomValues`) is a meaningful security improvement over the prior unsalted approach.

The one concern worth flagging is that SHA-256 is used for hashing security answers. SHA-256 is a general-purpose fast hash — it is not a password-hardening function (unlike bcrypt, scrypt, or Argon2). A determined attacker with database access could brute-force common security answers against SHA-256 hashes relatively quickly. For security questions (which tend to have low-entropy answers like maiden names or pet names), a KDF with a high work factor would be more appropriate. The current implementation is better than plaintext, and the salt does help, but it is not as strong as industry-recommended password storage.

**Strengths:**
- No secrets in source code; env-var-only configuration with fail-fast validation
- CSP headers restrict external script/style/connection sources
- CORS restricted to `SITE_URL` in all Edge Functions
- Per-user random salt added to security answer hashing

**Issues:**
- Security answers hashed with SHA-256 (a fast hash), not a KDF like bcrypt or Argon2 — low-entropy answers are more vulnerable to brute-force than they should be
- `ErrorBoundary` displays `error.message` in the UI — could expose stack traces or internal paths if an unexpected error includes sensitive context

### 9. DevOps & CI/CD — C+ (70/100)

*What this measures: Whether there is automated infrastructure for building, testing, and deploying the application reliably.*

The developer workflow tooling is solid: Husky manages three Git hooks (pre-commit runs Prettier via lint-staged, commit-msg enforces commitlint, pre-push runs the full test suite). Vitest coverage thresholds provide a soft gate on code quality. Playwright E2E is configured and runnable. Supabase migrations provide a versioned, reproducible database schema.

The missing piece is a cloud CI pipeline. There are no GitHub Actions workflows, no `.github/` directory, and no equivalent CI config for any other platform (GitLab CI, CircleCI, etc.). This means tests only run if a developer remembers to run them locally (or if the Husky pre-push hook is not bypassed). A missed `--no-verify` flag would let broken code reach main. For a project with a published PWA, a CI pipeline that runs `npm test`, `npm run typecheck`, `npm run lint`, and `npm run build` on every push would provide a meaningful safety net.

The `DEPLOYMENT_CHECKLIST.md` documents the manual deployment steps, which is valuable, but manual processes can be skipped under pressure.

**Strengths:**
- Husky enforces Prettier, commitlint, and tests at the Git hook level
- Coverage thresholds in `vite.config.ts` prevent regressions below a minimum bar
- 26 Supabase migrations provide a reproducible, versioned database schema

**Issues:**
- No CI/CD pipeline (no `.github/workflows/`, no equivalent) — tests only run locally
- Deployment is a documented manual checklist, not an automated pipeline
- Coverage thresholds are enforced only on `npm run test:coverage`, not automatically on every push in CI

## Top 5 Priorities

1. **Split `exerciseService.ts` into domain modules** — At 865+ lines covering 7 distinct domains, this file is the single biggest maintainability risk. Splitting into `weekService.ts`, `questionService.ts`, `answerService.ts`, and `trackerService.ts` would make each file independently testable, greppable, and understandable. Medium effort, high long-term payoff.

2. **Add a GitHub Actions CI pipeline** — A workflow that runs `typecheck`, `lint`, `test:coverage`, and `build` on every push/PR would catch regressions that Husky misses (bypassed with `--no-verify`, CI machines, etc.). Low effort, immediate safety improvement.

3. **Add unit tests for the service layer and custom hooks** — `exerciseService.ts` and `legalService.ts` are excluded from coverage. `useAuth` and `useLegalStatus` have no tests. These are the highest-risk code paths in the app (data mutations, auth state, legal gating) and exactly where bugs are most costly. Medium effort, high value.

4. **Upgrade security answer hashing from SHA-256 to Argon2/bcrypt** — The current SHA-256 + salt is better than nothing, but security answers (pet names, maiden names) have low entropy. A server-side KDF via a Supabase Edge Function would make brute-forcing impractical. Medium effort, meaningful security improvement for a journaling app with personal data.

5. **Clean up development artifacts** — Remove the four CSV seed files from `src/data/`, the `New folder/` directory, fix the doubled `.sql.sql` migration filename, and update `appConfig.ts` with real branding. Low effort, reduces confusion for future contributors.

## What's Working Well

**Service layer discipline is exemplary.** No component in the codebase reaches directly into Supabase. Every data operation goes through a named function in the service layer, which makes the data contract explicit and the components free of infrastructure concerns. This pattern is consistently applied across all 29 components.

**The timezone module is a small gem.** `src/lib/timezone/` is a self-contained, well-structured utility with proper separation between data (`data/`), types (`types.ts`), and utilities (`utils/`), plus a clean barrel export. It could be lifted out of this project and published as a standalone package without modification.

**Active architectural improvement is clearly happening.** The CHANGELOG shows genuine code quality investment — Phase 1 added comprehensive error handling, Phase 2 extracted hooks from App.tsx, Phase 3 optimized queries and hardened security. The code reflects these improvements. This is not a project where technical debt is accumulating passively; it's being actively managed.

**The documentation suite is outstanding.** Having 11 focused documentation files, each covering a specific dimension of the system, is a hallmark of a developer who understands that code is read far more than it is written. The `FRONTEND.md` component catalog, in particular, is something many much larger teams don't produce.
